## Overview

Complete rebuild of split testing functionality from the ground up.

## Changes Made

### New API Endpoint: `/api/split-test`
- Dedicated endpoint for A/B testing
- Handles variant assignment (50/50 random)
- Stores assignments in memory (session-based)
- Returns appropriate popup ID for the assigned variant
- Full CORS support

### Frontend Updates
- New `handleSplitTest()` method in GCModal
- Calls dedicated `/api/split-test` endpoint
- Shows assigned variant popup
- Maintains assignment consistency per session

### API Flow
1. User clicks split test button (ID: `split-{testId}`)
2. Frontend calls `/api/split-test?id={testId}`
3. API checks Control Board for test configuration
4. If completed: returns winner popup ID
5. If running: assigns variant A or B (50/50), stores assignment, returns popup ID
6. Frontend shows the assigned popup

## Testing

### To Test Split Testing:
1. Create split test in admin dashboard
2. Add split test button to page:
   ```html
   <button id="split-your-test-id">Click Me</button>
   ```
3. Include GC Modal script
4. Click button - should show Variant A or B randomly
5. Click again - should show SAME variant (assignment stored)

## Known Limitations

- Assignment storage is in-memory (lost on server restart)
- For production, should use Redis or database
- No conversion tracking yet
- No deduplication by email yet
